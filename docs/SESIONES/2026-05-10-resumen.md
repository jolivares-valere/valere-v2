# Sesión 2026-05-10 — FV Alta Manual Credenciales + Split-Table Security

## Qué se hizo

### Problema de partida
El módulo FV solo tenía datos de demo (fixtures). Para poder operar de verdad con portales FV reales hacía falta:
- Un formulario para introducir credenciales reales (usuario + contraseña)
- Almacenamiento seguro de esas credenciales
- Asignación manual de plantas detectadas a clientes existentes en el CRM

### Arquitectura de seguridad implementada

**Problema crítico detectado durante la sesión:** la primera implementación escribía `password_enc` directamente desde el frontend. Bloqueado por revisión de seguridad del usuario.

**PostgREST REVOKE SELECT(column) NO funciona:** PostgREST genera SELECT * internamente, revocar columnas individuales rompe todas las queries. Por eso el split-table era la única solución correcta.

**Solución — split-table model:**

    fv_credenciales         → datos operativos (admin/master pueden leer desde frontend)
    fv_credenciales_secret  → password_enc + session_cookies (SOLO service_role)

Tres capas de protección sobre fv_credenciales_secret:
1. RLS habilitado sin ninguna policy → authenticated obtiene []
2. REVOKE ALL ON fv_credenciales_secret FROM authenticated, anon
3. Frontend nunca llama a esta tabla — solo Edge Function y Python sync (service_role)

### Python sync — cambio de columnas

Antes (se rompe tras migración):
    cred["password_enc"]        # KeyError
    cred.get("session_cookies") # None siempre

Después (compatible con fv_credenciales_secret):
    secret = cred.get("_secret")  # dict del JOIN, None si sin fila
    secret.get("password_enc")    # seguro, con manejo de None
    secret.get("session_cookies") # idem

## Rollback SQL (si algo falla tras la migración)

    -- Restaurar columnas en fv_credenciales
    ALTER TABLE public.fv_credenciales
      ADD COLUMN IF NOT EXISTS password_enc TEXT,
      ADD COLUMN IF NOT EXISTS session_cookies TEXT,
      ADD COLUMN IF NOT EXISTS cookies_expires_at TIMESTAMPTZ;

    -- Recuperar datos desde fv_credenciales_secret
    UPDATE public.fv_credenciales fc
    SET
      password_enc       = s.password_enc,
      session_cookies    = s.session_cookies,
      cookies_expires_at = s.cookies_expires_at
    FROM public.fv_credenciales_secret s
    WHERE s.credencial_id = fc.id;

    -- Eliminar tabla secreta
    DROP TABLE IF EXISTS public.fv_credenciales_secret;

    -- Revertir fv_planta (solo si no había datos nuevos)
    ALTER TABLE public.fv_planta
      DROP COLUMN IF EXISTS cups_id,
      DROP COLUMN IF EXISTS sync_enabled,
      DROP COLUMN IF EXISTS nombre_fusionsolar,
      DROP COLUMN IF EXISTS nombre_interno;

## Decisiones importantes

- No merge a main hasta que la ventana de despliegue se haga completa (migración + EF + Python juntos)
- El módulo FV con fixtures funciona en producción hoy — el cambio no afecta nada hasta aplicar migración
- --check-secrets es el comando de verificación post-migración antes de cualquier sync real
- La clave FV_ENCRYPTION_KEY debe ser la misma en Supabase Secrets y en el .env del conector Python local

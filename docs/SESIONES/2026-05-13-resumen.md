# Sesión 2026-05-13 — FV Hardening: EU5 Fix + Overflow Guards + Audit Scaffold

## Resumen ejecutivo

Sesión larga enfocada en diagnosticar y corregir por qué el workflow `fv-sync` fallaba en producción con `get_daily_kpi`. Se identificaron tres bugs independientes, se corrigieron todos, se redujo la frecuencia de cron y se diseñó el sistema de auditoría `fv_sync_audit`. El workflow sigue bloqueado porque GitHub Actions ejecuta los crons desde `main` (código viejo) — el fix está listo en un script de merge.

---

## Qué se hizo

### 1. Fix EU5 — `get_daily_kpi` con campos correctos (`45a7c3a`)

**Diagnóstico**: El portal EU5 de FusionSolar usa nombres de campo distintos al portal global:
- Identificador de planta: `"dn"` (ej: `"NE=135347362"`) vs `"stationCodes"` (global)
- Timestamp: `"queryTime"` + `"timeZone": 2` vs `"collectTime"` (global)

El código enviaba `stationCodes`+`collectTime` → EU5 respondía HTTP 503.

**Fix**: Detectar EU5 por `station_code.startswith("NE=")`. Si es EU5, usar los campos correctos. Aplicado en ambas implementaciones del cliente:
- `WebAuthClient.get_daily_kpi` (línea ~388)
- `StorageStateClient.get_daily_kpi` (línea ~879) — también añadido retry x3 con backoff 5s/10s

### 2. Fix overflow — HOTEL SIERRA LUZ (`45a7c3a`)

**Diagnóstico**: FusionSolar devolvía un valor de `potencia_max_kw` fuera del rango PostgreSQL `numeric(10,2)` para HOTEL SIERRA LUZ. El insert en `fv_kpi_diario` fallaba con `numeric field overflow`. La causa probable es un bug de caché en el portal EU5 (devuelve suma global en vez de dato de la planta).

**Fix**: Guards en el backfill loop de `sync_job.py`:
```python
if not (0 <= energia <= 999_999): energia = 0.0
if not (0 <= potencia_max <= 99_999): potencia_max = 0.0
```

Estado DB verificado: 0 filas en `fv_kpi_diario` para HOTEL SIERRA LUZ (datos limpios, no hay corrupción).

### 3. Cron diario + anti-solapamiento (`45a7c3a`)

- Cron: `0 * * * *` (24x/día) → `0 7 * * *` (1x/día a las 07:00 UTC)
- Añadido bloque `concurrency.group: fv-sync` + `cancel-in-progress: false` en `fv-sync.yml` para evitar runs solapados

### 4. `fv_sync_audit` — scaffold de auditoría (NO COMMITADO aún)

Diseñado e implementado el sistema de observabilidad por planta:

**`supabase/migrations/20260513_fv_sync_audit.sql`** (creado, pendiente aplicar):

- Tabla `fv_sync_audit` con run_id, credencial_id, planta_id, fecha_dato, endpoint, ok, error_tipo, intentos, filas_insertadas, ms_elapsed

**`sync_job.py`** (modificado, pendiente commit con `COMMIT_FV_AUDIT.ps1`):
- Helper `_audit_record()`: insert en `fv_sync_audit`, no bloqueante (wrapped en try/except)
- Helper `_classify_error()`: taxonomía automática de errores
- `run_id = str(_uuid_mod.uuid4())` al inicio de `sync_credencial()` — agrupa registros de una ejecución
- Backfill instrumentado: audit OK (con filas_insertadas) y FAIL (con error_tipo clasificado)

**Diseño deliberado**: `_audit_record()` no falla si la tabla no existe aún — permite desplegar instrumentación antes de aplicar el SQL. El log mostrará `_audit_record error (no bloqueante): relation "fv_sync_audit" does not exist` como warning esperado.

### 5. Investigación del bug "workflow corre desde main"

**Síntoma**: 2 intentos de lanzar el workflow desde `feature/fv-operational-redesign` produjeron `KeyError: 'password_enc'` — error que solo existe en el código de `main`.

**Causa confirmada**: Los scheduled crons de GitHub Actions siempre usan el branch por defecto (main), independientemente del desplegable "Use workflow from" en la UI. Los `workflow_dispatch` manuales también pueden silenciosamente defaultear a main.

**Evidencia**: `refs/remotes/origin/main` en los logs de ambos runs. El código de `main` tiene `cred["password_enc"]` en línea 248; `feature/` tiene `secret.get("password_enc")` via `load_fv_credentials_with_secrets()`.

**Solución**: Merge `feature/fv-operational-redesign` → `main` directamente. Script preparado: `~/.claude/MERGE_FEATURE_TO_MAIN.ps1`.

## Decisiones importantes

1. **Merge directo feature→main** en vez de parchear `main` para añadir `_load_credential_secret()` (sugerido por ChatGPT/Gemini). Razonamiento: el feature branch ya tiene la implementación correcta y completa; añadir parches al código viejo de main sería acumular deuda técnica sin justificación.

2. **Audit no bloqueante** — `_audit_record()` completamente aislado en try/except. Si la tabla `fv_sync_audit` no existe, el sync continúa normalmente. Esto permite desplegar el código instrumentado antes de aplicar el SQL.

3. **Cron a las 07:00 UTC** (no 08:00) para que los datos diarios estén disponibles en el CRM cuando el equipo empieza a trabajar (09:00 hora España en verano).

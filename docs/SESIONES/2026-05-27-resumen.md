## Hallazgos críticos

### El módulo está al ~70% construido
Lo que el briefing trataba como "construir desde cero" es en realidad "completar y conectar":
- `comercializadoras` + `comercializadora_ofertas` → CRUD admin completo + importador XLSX masivo.
- `AnalisisPage` ya implementa el comparador end-to-end (lee facturas históricas, simula N ofertas con `calculateSimulatedInvoice`, rankea por ahorro, guarda en `proposals`).
- `PropuestasEnergiaPage` visualiza propuestas con KPIs + export CSV.
- Patrón Edge Function Gemini ya desplegado (`chat-consultor`).
- Sistema de roles + RLS granular ya en pie.

### Lo que realmente falta (delta)
1. Capa de ingesta automática (Make → webhook → tabla `tariff_documents`).
2. Capa de extracción IA (Edge Function `tariffs-extract` clonando patrón `chat-consultor`).
3. Versionado de ofertas (hoy el upsert sobreescribe — añadir `valid_from`/`valid_to`/`status`/`version` + RPC `publish_oferta_with_versioning`).
4. Bandeja de tarifas pendientes (humano-en-el-bucle).
5. Logo de comercializadora (campo no existe).
6. Generador PDF de propuesta (bloqueado por NEG-B).
7. Borrador de email con aprobación manual.

### Diferenciación crítica de motores de cálculo
El repo tiene **dos motores**:
- `calculateInvoiceEstimate` (datadis_supply_price_terms) → estima la factura actual del cliente. Usado en `SupplyDetailPage` tab "Factura Teórica".
- `calculateSimulatedInvoice` (`/core/energia/calculator`) → simula factura con una oferta concreta. Usado en `AnalisisPage` para comparador.

Conviene NO mezclarlos. A futuro pueden unificarse en una librería con dos puntos de entrada, pero no es prerrequisito del módulo.

### Trampa de nomenclatura
`src/core/types/database_canonical_2026-04-26.ts` muestra `retailers/retailer_offers`. Las migraciones `20260425_*` y `20260426_*` renombraron a `comercializadoras/comercializadora_ofertas` pero los tipos no se regeneraron. **Cualquier código nuevo del módulo debe usar los nombres en español.** Regenerar tipos es prerrequisito de Fase 0.

## Decisiones técnicas tomadas

| Decisión | Resolución |
|---|---|
| FastAPI vs Edge Functions | **Edge Functions** (alineación con stack actual, 8 funciones ya desplegadas, patrón Gemini reutilizable) |
| Dedup primario | **Identificador lógico** (`comercializadora + producto + access_rate + valid_from`). SHA256 sólo como atajo barato pre-IA. |
| Consumo cliente | **Datadis principal** (vía tabla `facturas` ya sincronizada). Excel como fallback para prospectos sin Datadis. |
| Versionado ofertas | **Estado en la misma tabla** (`status='superseded'` + `valid_to` + `superseded_by`), no tabla history aparte. |
| Bandeja validación | **Tab dentro de AdminPage** (no pantalla nueva). |
| Renderizado PDF | **`@react-pdf/renderer` cliente** para MVP (más control visual, fácil iterar). |
| Generador email | **Resend** (ya conectado para auth). |
| Modelo Gemini | **`gemini-2.5-flash-lite`** o `gemini-3.1-flash-lite` para extracción (lee PDF nativo, coste mínimo). Posible mini-refactor de `chat-consultor` para alinear modelos. |

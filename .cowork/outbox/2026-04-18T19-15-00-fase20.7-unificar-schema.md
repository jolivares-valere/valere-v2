# FASE 20.7 — Unificar schema Supabase (Calculadora → CRM)

## Contexto

Tras la fusión de los dos módulos (CRM + Calculadora) en arquitectura
feature-based, el schema de BD todavía mantiene tablas duplicadas de la
Calculadora (`clients`, `supply_points`, `invoice_history`) que no se han
migrado a las tablas canónicas del CRM (`empresas`, `cups`, `facturas`).

**Canónica de perfiles**: `user_profiles` (Calculadora, plural). `users_profile`
(CRM, singular) es legacy. Ambas tienen 1 fila — migración trivial.

## Tareas

Crear 3 migraciones SQL **idempotentes** en `supabase/migrations/`:

### 1. `20260418_fase20.7.a_users_profile_merge.sql`

```sql
-- Migrar la fila de users_profile → user_profiles si no existe.
INSERT INTO public.user_profiles (id, email, full_name, role, approved, created_at)
SELECT up.id,
       u.email,
       up.nombre_completo,
       CASE up.rol
         WHEN 'admin' THEN 'master'
         WHEN 'jefe_equipo' THEN 'manager'
         WHEN 'comercial' THEN 'consultant'
         ELSE 'consultant'
       END,
       up.activo,
       up.created_at
FROM public.users_profile up
JOIN auth.users u ON u.id = up.id
ON CONFLICT (id) DO NOTHING;

-- Ajustar FKs que apunten a users_profile para que apunten a user_profiles.

```

### 2. `20260418_fase20.7.b_clients_to_empresas.sql`

```sql
-- Mapear clients.consultor_asignado (email TEXT) → empresas.comercial_id (UUID)
-- vía lookup en user_profiles.
INSERT INTO public.empresas (id, nombre, nif, email_principal, telefono_principal,
  direccion, cp, ciudad, provincia, comercial_id, created_at)
SELECT c.id,
       COALESCE(c.nombre_comercial, c.razon_social, 'Sin nombre'),
       c.nif,
       c.email,
       c.telefono,
       c.direccion,
       c.cp,
       c.ciudad,
       c.provincia,
       (SELECT id FROM public.user_profiles WHERE email = c.consultor_asignado LIMIT 1),
       c.created_at
FROM public.clients c
ON CONFLICT (id) DO NOTHING;
-- (No DROP clients todavía — esperar confirmación de datos migrados OK.)
```

### 3. `20260418_fase20.7.c_supply_points_to_cups.sql`

```sql
-- Mapear supply_points → cups (ya existe tabla cups en CRM).
-- supply_points.client_id → cups.empresa_id (ya migrada en 20.7.b).
INSERT INTO public.cups (id, empresa_id, codigo_cups, direccion_suministro,
  distribuidor, estado, created_at)
SELECT sp.id,
       sp.client_id,
       sp.cups,
       sp.direccion,
       sp.distribuidora,
       'activo',
       sp.created_at
FROM public.supply_points sp
WHERE NOT EXISTS (SELECT 1 FROM public.cups c WHERE c.codigo_cups = sp.cups);
```

### 4. `20260418_fase20.7.d_invoice_history_rename.sql`

```sql
ALTER TABLE IF EXISTS public.invoice_history RENAME TO facturas;

-- Añadir FK a cups si la columna supply_point_id existe y no tiene FK.
DO $$
BEGIN
  IF EXISTS (SELECT 1 FROM information_schema.columns
             WHERE table_name='facturas' AND column_name='supply_point_id')
  THEN
    ALTER TABLE public.facturas
      ADD CONSTRAINT facturas_cups_fkey
        FOREIGN KEY (supply_point_id) REFERENCES public.cups(id) ON DELETE CASCADE
      NOT VALID;
  END IF;
END $$;
```

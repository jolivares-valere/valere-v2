## Nota sobre migration aplicada

La migration `20260424_fase2_rag_assistant_setup.sql` se ha aplicado directamente al proyecto Supabase CRM vía MCP (no vía `supabase db push`). Eso significa que en Supabase ya está la tabla, pero en el historial local de migrations Supabase CLI no está registrada.

**Para sincronizar** (próxima sesión):

```powershell
cd $HOME\valere-v2
npx supabase db remote commit  # o similar según tu workflow
```

O simplemente el SQL está versionado en `supabase/migrations/` y si alguien clona el repo y hace `supabase db push`, Supabase detecta que ya está aplicada y la marca como tal.

## Reglas aprendidas en este sprint

- **Aplicar migrations vía MCP sin ejecución de pg_dump previo está bien para setups nuevos (no-destructivos)**, pero siempre versionar el SQL en el repo para trazabilidad.
- **Gramática DDL idempotente** (`create ... if not exists`, `drop policy if exists`) es obligatoria para migrations que puedan re-ejecutarse.
- **Security invoker + `set search_path`** en funciones PL/pgSQL es la forma correcta de proteger contra hijacking de search_path (se venía avisando en los advisors de Supabase).
- **Escribir docs de ayuda desde el código, no desde la memoria**: inspeccionar los valores reales del código (p.ej. las 8 etapas de oportunidades, los 6 tipos de actividad) hace la doc mucho más precisa.
